```typescript
  // D-05 / D-06 / DRAFT-04: intercept in-app navigation while the dialog is open.
  // Predicate is synchronous — it just decides whether to block. The actual save
  // lives in the effect watching `blocker.state`.
  //
  // Gates:
  //   1. open === true — otherwise the dialog isn't visible, nothing to save
  //   2. selectedCustomer !== null — D-03/D-04 gate (no empty drafts)
  //   3. pathname changed — skip blocking on hash/search-only changes
  //   4. nextLocation.pathname !== '/login' — Pitfall 8: don't intercept the
  //      auth-driven redirect when the axios 401 interceptor logs the user out
  const blocker = useBlocker(
    ({ currentLocation, nextLocation }) =>
      open &&
      !!selectedCustomer &&
      currentLocation.pathname !== nextLocation.pathname &&
      nextLocation.pathname !== '/login'
  );

  useEffect(() => {
    if (blocker.state !== 'blocked') return;

    // D-05: fire the save but DO NOT await — navigation must proceed immediately.
    // performDraftSave already handles its own errors (destructive toast) — any
    // rejection here is best-effort by design.
    void performDraftSave();

    // Proceed immediately so the sidebar link feels instant.
    blocker.proceed?.();
    // Intentionally depend on blocker.state only — re-running on blocker identity
    // changes would fire proceed on every render.
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, [blocker.state]);
```

### Why this is safe under Plan 03's handleCloseRequested

- User clicks Abbrechen → `handleCloseRequested` → `onClose()` sets parent `dialogOpen=false` → component re-renders with `open=false` → blocker predicate returns false next time. No interference.
- User clicks sidebar link with dialog open → blocker intercepts → effect fires save + proceeds. The dialog doesn't get a close event (user didn't click Abbrechen or X), so `handleCloseRequested` is NOT called — only the blocker's save runs. This avoids a double-POST.
- User clicks X button → Radix fires `onOpenChange(false)` → `handleCloseRequested` runs → `onClose()` → no navigation, so blocker never fires. Close-save path only.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| router navigation events → blocker predicate | untrusted URL transitions; predicate is synchronous and allowlist-gated |
| effect firing fire-and-forget save | untrusted race: user may navigate again before save resolves |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02B-01 | Denial of Service | Blocker stuck in 'blocked' state after logout | mitigate | Predicate explicitly returns false when `nextLocation.pathname === '/login'` (Pitfall 8); blocker.proceed?.() called in the effect so state clears |
| T-02B-02 | Information Disclosure | Blocker intercepts a legitimate auth-driven redirect and leaks form data via the save call against an expired token | accept | Existing axios 401-interceptor refreshes the token transparently; if refresh fails, the save rejects silently (fire-and-forget) — no data leak, just a best-effort loss documented in UI-SPEC |
| T-02B-03 | Spoofing | Forged navigation events via history.pushState manipulation | accept | `useBlocker` is the official supported API; it handles history stack, forward/back, and programmatic navigate() uniformly — a malicious actor in the browser session already controls the app fully |
| T-02B-04 | Tampering | Blocker predicate returns stale values due to closure over old state | mitigate | Predicate reads `open` and `selectedCustomer` from the closure; React re-invokes the component on every state change so a fresh predicate is registered each render — consistent with React Router docs |
| T-02B-05 | Elevation of Privilege | Draft save during nav-away using stale role | mitigate | JWT-based role enforcement on backend `requireRole('owner', 'admin', 'reception')` — client-side ref values cannot bypass server gate |
| T-02B-06 | Repudiation | User claims navigation was blocked | accept | UI-SPEC explicitly documents silent nav save — failure toast may not render before navigation, accepted for Phase 2; Phase 4 improves feedback |

All threats `mitigate` except T-02B-02 (delegated to interceptor), T-02B-03 (not a realistic threat model), and T-02B-06 (UI-SPEC accepts the limitation).
</threat_model>
