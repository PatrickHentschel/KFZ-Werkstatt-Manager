```typescript
  async deleteDraft(tenantId: string, id: string): Promise<void> {
    const existing = await db.query.invoices.findFirst({
      where: and(eq(invoices.id, id), eq(invoices.tenantId, tenantId)),
    });
    if (!existing) throw errors.notFound('Draft');
    if (existing.status !== 'draft') {
      throw errors.badRequest('Only drafts can be deleted via this endpoint');
    }

    // invoice_items has ON DELETE CASCADE on invoiceId (verified in schema) —
    // deleting the parent invoice removes its items in the same statement.
    await db
      .delete(invoices)
      .where(and(eq(invoices.id, id), eq(invoices.tenantId, tenantId)));
  }
```

- No manual cascade — Phase 1 RESEARCH confirmed `onDelete: 'cascade'` on the `invoice_items.invoiceId` FK.
- No tenant-counter interaction (draft numbers live off the sequence — Phase 1 architectural decision).
- Return type is `Promise<void>`; route handler will use `reply.code(204).send()`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API (DELETE /draft/:id) | untrusted UUID param + untrusted JWT; RLS-enforced tenant scope |
| service → database | parameterised SQL via Drizzle; no raw concatenation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | DELETE /draft/:id route | mitigate | `fastify.requireRole('owner', 'admin', 'reception')` preHandler — technicians returned 403 by existing `requireRole` plugin; asserted by `grep -c "requireRole('owner', 'admin', 'reception')"` == 3 in routes file |
| T-02-02 | Information Disclosure | deleteDraft service cross-tenant lookup | mitigate | Explicit `AND tenant_id = $1` in both the SELECT (findFirst where clause) and the DELETE where clause; RLS GUC `app.current_tenant_id` as defence in depth (existing plugin) |
| T-02-03 | Tampering | deleteDraft bypassing status check to delete non-drafts | mitigate | Service throws `errors.badRequest('Only drafts can be deleted via this endpoint')` when `existing.status !== 'draft'` |
| T-02-04 | Spoofing | DELETE called with forged/expired JWT | mitigate | Plugin-level `addHook('preHandler', fastify.authenticate)` at line 65 runs on every route including the new DELETE — unchanged from Phase 1 |
| T-02-05 | Information Disclosure | 404 vs 403 oracle leak (existence probing) | accept | Service returns the same `errors.notFound('Draft')` for both "does not exist" and "exists in another tenant"; no timing difference to discriminate — matches Phase 1 pattern |
| T-02-06 | Tampering | SQL injection via `:id` path param | mitigate | Drizzle parameterised queries (`eq(invoices.id, id)`); no `sql\`\`` template string construction with user data in this path |
| T-02-07 | Denial of Service | Flood of DELETE calls | accept | Existing Fastify rate-limit plugin applies to every route at the app level; no phase-specific mitigation |
| T-02-08 | Information Disclosure | DraftInvoicePayload leaking server-only fields (tenantId, status, invoiceNumber) | mitigate | Frontend type does NOT include tenantId/status/invoiceNumber; even if client sent them, backend Zod `draftInvoiceSchema` does not list them so they are stripped |

All threats are `mitigate` except T-02-05 (information-disclosure-via-404) and T-02-07 (rate-limiting delegated to existing plugin) which are `accept` with documented rationale.
